3Decision: why the PoC rides Greengrass, not plain IoT Core

The plan document recommended plain IoT Core for V1 — written from a starting-from-zero assumption. By the time the PoC started, Greengrass was already installed, enrolled, and proven on all three PCs (it's how app code deploys now), which flips the trade-off: the listed Greengrass cons ("another runtime to install", "slower path to V1") were already sunk cost.

ConcernPlain IoT Core scriptGreengrass component (chosen)
AWS connectionown MQTT client, per-device certs to manage in the scriptfree — messages hand off to the nucleus's existing secured connection; the watchman contains zero connection/cert code
Getting it onto PCsown installer + copy mechanismfree — versioned remote deployment, same as app code
Keeping it aliveNSSM / Windows service to buildfree — Greengrass supervises and restarts it
Lock-in—minimal: topics and payloads are plain IoT Core MQTT either way; swapping to the Device SDK later changes ~20 lines
The important invariant Everything above the transport is exactly as specified in the plan: same topic scheme, same command/result payload shape, same watchman responsibilities. Greengrass only answers "how do messages travel and how does code arrive" — the questions the fleet already answered for the app.

5What it costs at scale

Three meters run (prices verified July 2026: Greengrass, IoT Core). Per PC per month, at the PoC's 30-second heartbeat cadence:

MeterPricePer PC / month
Greengrass core device$0.16/month flat per enrolled device$0.16
Connection time$0.08 per million minutes~$0.003 (connected 24/7)
Messages$1 per million~$0.09 (30s heartbeats)
Shadow updates$1.25 per million~$0.11 (every heartbeat)

≈ $0.36 per PC per month → 3 PCs ≈ $1/month · 100 PCs ≈ $36/month (~£27). Code deployments barely register (a 2.9 MB release to 100 PCs is cents; S3 storage is pennies).

Two levers worth knowing Over half the per-device cost is our own reporting cadence — a slower pulse and change-only shadow updates roughly halve it. And the real future cost centre is centralised logging (CloudWatch bills per GB ingested): chatty logs from 100 PCs would dwarf everything above, so log levels/sampling are a design-time decision for Phase 3, not an afterthought.

6How a human finds out a machine is down

Nobody watches the stream. AWS is wired — once — to watch it and page someone. Both layers are serverless configuration, not services we run:

FailureDetected byWiring
PC offlineAWS notices its own connection to the device dropped (same signal as the console's Healthy status)Greengrass health event → EventBridge rule → SNS → email/Slack
PC online but app/watchman sickHeartbeats stop arriving — silence is the alarm; a dead process can't report itselfIoT Rule feeds a CloudWatch metric → alarm on "no data for 15 min" → SNS → email/Slack

On-call experience at 100 PCs: phone buzzes — "holly-047 silent 15 min" — investigate over Tailscale (VNC/SSH). This also sets the production heartbeat design: a slow pulse (~5 min) so silence is detectable, events when something actually happens ("app crashed, restarted it"), and a status command for on-demand detail — less traffic than the PoC's 30s demo firehose, better signal.

Effort & where it fits About an hour of AWS-side config, pennies to run. Belongs in Phase 3 with the rest of the cloud plumbing — and it's pure AWS-side wiring, making it a natural first CloudFormation template.

7Naming & grouping — proposed convention

The plan proposes holly-edge-{customer}-{site}-{number}. The problem: that bakes changeable facts into an unchangeable name. A thing's name is permanent (renaming = re-enrolling the device), but kiosks get redeployed — move a PC to another customer and its name lies forever. The standard fleet pattern splits the jobs:

ThingHoldsChangeable?
Namestable serial identity: holly-001, holly-002… (matches hostname + Tailscale name — three consoles, one identity)never — so keep it fact-free
Attributesbusiness facts: customer=vena, site=london-hq, role=receptionyes — one API call
Groupscollections: holly-fleet (all devices), per-customer / per-site groupsjoin / leave freely

How groups behave: a thing can be in multiple groups (up to 10); groups can nest into hierarchies; and dynamic groups can define membership by query ("every thing where customer = X") so they maintain themselves off the attributes. Everything the long name would have provided — "all of customer X's devices", "command every device at site Y" — comes from groups/attributes instead, with no re-enrolment tax.

Aiming deployments at facts ("all of company X's PCs")

Greengrass deployments accept dynamic thing groups as targets — and a dynamic group is a fact wearing a target costume: define it once as "everything where customer=vena" and deploy to it. Move a PC to another customer by changing one attribute; it automatically leaves one deployment audience and joins the other. Two clean shapes for what those deployments carry:

  • Same app everywhere, per-customer config (recommended default): holly-fleet ships the software; customer groups ship only a small config component or settings overrides. Different components → merges cleanly on the device.
  • Genuinely different app versions per customer: then there's no global software group — each customer group owns software for its devices, and every device sits in exactly one. More overhead; only if truly needed.
One rule to keep Any one component is only ever shipped to a device by one group. Deployments from multiple groups merge on a device and conflict if two groups ship the same component at different versions — the two shapes above are just the two ways of honouring that rule.
