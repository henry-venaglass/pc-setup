01

The two halves, and the line between them

Everything splits into two responsibilities. The cloud side — AWS IoT Greengrass — delivers the code and watches the fleet. The local Windows session — auto-logged-in as holly, running 24/7 — draws the pixels on screen. Greengrass owns delivery and only delivery; the local half is already built and doesn't change.

Cloud · AWS IoT Greengrass

Owns the management layer
  • Ships new versions of the holly app to the fleet (versioned, with per-device rollback)
  • Devices pull releases themselves — your Mac doesn't need to be on
  • Reports device connectivity & deployment health to dashboards / alerts
  • Per-device certificate identity & security auditing

Local · holly's Windows session

Owns what's on the screen
  • Auto-login as holly, 24/7 — set up by setup.ps1
  • watchdog.ps1 launches uv run holly at 08:30, closes it at 18:00, Mon–Fri
  • The Qt window picks monitor 2 and goes fullscreen (app concern)
  • The same watchdog relaunches the app if it crashes or hangs mid-day
session 0
Greengrass runs as a Windows service, and services live in session 0, isolated from the interactive desktop. A GUI launched from there will not appear on any monitor. That single fact is why the visible launch must always be local — under AWS or any other tool. Greengrass can deliver the app and confirm the device is alive; it must never be the thing that paints it on monitor 2. On this fleet, the thing that paints it is watchdog.ps1.
Worth knowing Greengrass is adopted from the start deliberately — having the fleet machinery in place before expanding beats retrofitting it after. Nothing local depends on it though: setup.ps1 and watchdog.ps1 are identical with or without the cloud layer, and the tailnet stays on every NUC for VNC and interactive SSH.
02

What the repo already covers

Two scripts in pc-setup carry the fleet — one you run per NUC, one you run per release — plus the watchdog loop, which lives inside setup.ps1 and gets written onto every PC.

setup.ps1
First-boot provisioning — the one script you run on a PC. Run once per NUC with -PCNumber xxx -AuthKey tskey-auth-… -AwsAccessKey … -AwsSecretKey … (the Tailscale/AWS keys are parameters, never committed; the holly password is deliberately hardcoded — wired-only LAN, physically controlled boxes, VNC only via tailnet). Renames the PC to HOLLY-xxx, configures auto-login as holly, debloats, installs TightVNC + OpenSSH server, hardens the kiosk, joins the tailnet, disables wifi/BT — then (§21) registers the watchdog launcher as a logon task, and (§22) enrols the device into Greengrass as holly-xxx in holly-fleet. Writes C:\code\setup-status.json.
publish.sh
Releases — the one script you run on the Mac. Zips the holly source, uploads it to S3, publishes a numbered Greengrass component version, and deploys it to holly-fleet. Devices pull it themselves — offline NUCs catch up when they reconnect, and a newly enrolled NUC receives the current version automatically. Replaced push.sh (scp over the tailnet).
watchdog.ps1
Runtime — fleet-managed. Lives in this repo; publish.sh packs it into every release, so it lands and updates at C:\code\holly\watchdog.ps1 on every PC — a watchdog fix rolls out fleet-wide exactly like app code, no per-PC SSH. On each PC it's kept running by a tiny never-changing launcher (C:\code\watchdog-launcher.ps1, installed by setup.ps1 §21 as the logon task). It runs continuously in holly's interactive session: launches uv run holly from C:\code\holly\projects\holly-local, enforces the 08:30–18:00 Mon–Fri window, restarts on crash or hang (with restart-storm backoff), restarts itself when a release replaces its file, logs to C:\Watchdog. This is the local half of the boundary — Greengrass delivers it but never runs it.
Also in the repo — manual fallbacks push.sh — the old scp deploy loop, kept as the emergency fallback for when AWS is unreachable or one PC needs code pushed by hand (./push.sh holly-002). ssh_script.ps1 — retrofits SSH onto a PC that doesn't have it (pre-§12B machines, or a Windows reinstall), which is what push.sh needs to reach it. Simple how-to for both: manual-tools.html.
03

Naming the fleet in AWS

You already have the AWS account the agent is hosted on, so there's no account or IAM setup — just fix the names so the AWS console lines up with the Tailscale console. Keep thing names identical to the tailnet machine names.

Regioneu-west-2 (London) — match wherever the agent is already hosted.
Thing nameUnique per device, same as its tailnet name: holly-001, holly-002, … (Windows hostname stays HOLLY-001; thing names lowercase to match Tailscale.) The one value that changes per NUC.
Thing groupThe fleet you deploy to: holly-fleet. Deploy once → every device in the group gets it.
Component userA local account, ggc_user, that Greengrass runs components as — not holly. Components only move files; they never touch the screen, so they shouldn't run as the kiosk user.
04

Enrol a NUC into Greengrass

"Connecting a device" isn't a button — installing the Greengrass Core software is the connection. The installer creates the Thing, generates the certificate, makes the device's IAM role, and drops it into holly-fleet in one shot. This is all automated as setup.ps1 §22 — it runs when you pass -AwsAccessKey/-AwsSecretKey. The steps below are what §22 does under the hood, and how to enrol an already-provisioned PC by hand.

  1. Install a JRE

    Greengrass Core runs on the JVM. §22 installs Amazon Corretto via winget:

    PowerShell (admin)
    winget install --id Amazon.Corretto.11.JRE --silent --accept-package-agreements --accept-source-agreements
# java lands at C:\Program Files\Amazon Corretto\*\bin\java.exe
  2. Create the component user

    Make the local ggc_user account (random password, never expires) and store its credential where the Greengrass service can read it. The service runs as SYSTEM, so cmdkey must run as SYSTEM too — that's what PsExec is for. Plain cmdkey in your admin shell stores it in the wrong account's vault and enrolment will look fine but components will fail to run.

    PowerShell (admin)
    # create the user Greengrass runs components as
net user ggc_user "<a-strong-password>" /add
Set-LocalUser ggc_user -PasswordNeverExpires $true
# store the credential in SYSTEM's vault (PsExec from Sysinternals)
.\PsExec.exe -accepteula -s cmd /c "cmdkey /generic:ggc_user /user:ggc_user /pass:<a-strong-password>"
# let deployments write the app code
icacls C:\code /grant "ggc_user:(OI)(CI)M"
  3. Provide temporary AWS credentials

    Set them as environment variables for this shell only, from your existing account. Remove them once the install finishes.

    PowerShell
    $env:AWS_ACCESS_KEY_ID="AKIA..."
$env:AWS_SECRET_ACCESS_KEY="..."
  4. Run the installer with auto-provisioning

    This registers the Thing, installs the certificate, joins the group, and sets Greengrass up as a Windows service that starts on boot. Only --thing-name changes per device.

    PowerShell
    java -Droot="C:\greengrass\v2" -Dlog.store=FILE `
  -jar .\GreengrassInstaller\lib\Greengrass.jar `
  --aws-region eu-west-2 `
  --thing-name holly-001 `
  --thing-group-name holly-fleet `
  --component-default-user ggc_user `
  --provision true `
  --setup-system-service true

# then clear the provisioning creds:
Remove-Item Env:\AWS_ACCESS_KEY_ID, Env:\AWS_SECRET_ACCESS_KEY -ErrorAction SilentlyContinue
  5. Confirm it's "Healthy"

    The NUC opens a persistent MQTT connection and appears under Greengrass → Core devices as Healthy. Because it joined holly-fleet, it automatically pulls the group's current deployment — the app code arrives in C:\code\holly within a few minutes, no push needed.

Credential hygiene — scaled for a real fleet Don't provision with a personal admin key. Create a dedicated IAM user (e.g. greengrass-provisioner) with AWS's documented "Minimal IAM policy for installer to provision resources", use its access key for all enrolments, and deactivate the key in IAM between provisioning sessions. setup.ps1 already wipes the keys from each PC after enrolment. When enrolment volume grows (batches of PCs, other people provisioning), upgrade to fleet provisioning with claim certificates — AWS's Tailscale-authkey-style flow where the USB stick carries only a narrow claim certificate that devices exchange for their own identity, and account keys never touch a PC at all. It slots into §22 without changing anything else in the architecture.
Verify Installer flags, the Java version, and Windows service behaviour shift between Greengrass v2 releases. Check the current AWS docs before folding this into setup.ps1.
05

Schedule & self-heal: watchdog.ps1

An earlier draft of this runbook used two Task Scheduler jobs (start 08:30, stop 18:00) plus a separate watchdog. The repo collapsed all three into one script, and that's the better design here: watchdog.ps1 is a single loop that owns the whole window. Inside 08:30–18:00 on weekdays it makes sure uv run holly is running and relaunches it on crash or hang; outside the window (and all weekend) it closes the app gracefully, escalating to a process-tree kill if needed.

Launchuv run holly in C:\code\holly\projects\holly-local — a command line, not an .exe, which is why kill-by-image-name would be wrong here. The watchdog tracks the exact PID it started.
Window08:30–18:00, Mon–Fri only (parameters -StartTime / -EndTime).
CrashPolls every 3s; relaunches with the exit code logged.
HangOptional -DetectHang: an unresponsive window gets a 30s grace period, then a tree-kill and relaunch.
StormsMore than 5 restarts in 60s → back off 60s, so a broken build can't spin the CPU all day.
LogsC:\Watchdog\qtbrowser-watchdog.log — this file is also the natural heartbeat source (see 08).
Fresh PCIf C:\code\holly hasn't arrived yet (first Greengrass deployment still downloading), the launcher and watchdog wait for it rather than exiting — so setup order never matters.
Self-updateThe watchdog ships inside every release (publish.sh packs it in). When a deployment replaces C:\code\holly\watchdog.ps1, the running watchdog notices its own file changed, closes the app, and exits; the launcher immediately starts the new version, which relaunches the app. A watchdog fix reaches 100 PCs the same way an app fix does.

How it starts

Two layers, split by how often they change. setup.ps1 §21 installs a tiny launcher (C:\code\watchdog-launcher.ps1) and registers it as the Holly-Watchdog logon task, running interactively as holly so anything it launches paints on the real desktop. The launcher never changes — its whole job is "wait for C:\code\holly\watchdog.ps1 to exist, run it, rerun it if it exits". The watchdog (all the actual logic) arrives via deployments, so it's never frozen into any one PC. For PCs provisioned before §21 existed, copy the launcher out of setup.ps1 §21 to C:\code\watchdog-launcher.ps1 and run the same registration by hand:

PowerShell (admin) — only needed on pre-§21 PCs
$action = New-ScheduledTaskAction -Execute "powershell.exe" `
  -Argument '-NoProfile -WindowStyle Hidden -ExecutionPolicy Bypass -File C:\code\watchdog-launcher.ps1'

$trigger = New-ScheduledTaskTrigger -AtLogOn -User "holly"

# interactive logon as the auto-logged-in user = windows it spawns are visible
$principal = New-ScheduledTaskPrincipal -UserId "holly" -LogonType Interactive

$settings = New-ScheduledTaskSettingsSet -ExecutionTimeLimit ([TimeSpan]::Zero) `
  -RestartCount 3 -RestartInterval (New-TimeSpan -Minutes 1)

Register-ScheduledTask -TaskName "Holly-Watchdog" `
  -Action $action -Trigger $trigger -Principal $principal -Settings $settings -Force
Why the settings block matters Task Scheduler kills tasks after 72h by default — -ExecutionTimeLimit zero lets the launcher run forever, and the restart settings relaunch the launcher itself if it ever dies. To change the watchdog: edit watchdog.ps1 in the repo and run ./publish.sh — every PC picks it up like an app release. No SSH, no per-PC work.
06

Qt browser → fullscreen on monitor 2

Where the window appears is an app concern, and it belongs in the holly app itself (the Qt browser that uv run holly starts) — the app knows its own window and when it's ready, so it's far more reliable than any external nudging. Find the second screen, move the window onto it, go fullscreen.

Python · PySide6 / PyQt
from PySide6.QtGui import QGuiApplication

screens = QGuiApplication.screens()
# index 1 = second monitor; fall back to primary if only one
target  = screens[1] if len(screens) > 1 else screens[0]

window.setScreen(target)
window.setGeometry(target.geometry())
window.showFullScreen()
Tip Monitor order isn't guaranteed to match physical layout. If "screen 1" lands on the wrong panel, select by the target screen's known geometry/position rather than its list index.
07

Pushing code: publish.sh

A release is ./publish.sh from the Mac: it zips the holly source, uploads it to S3, publishes a numbered Greengrass component version, and deploys it to holly-fleet. (./publish.sh 1.4.0 to pin a version; otherwise the patch number auto-bumps.) What that buys over the old scp loop:

VersionedEvery release is a numbered component version — you always know what's on which NUC.
Auto-rollbackA device that fails a deployment reverts to the previous working version.
Staged (optional)Not enabled — releases go to all devices at once, which fits how this fleet is used (code lands once; day-to-day changes happen in the cloud). If frequent pushes ever become the norm, add an --iot-job-configuration rollout/abort block to publish.sh — Mac-side only, works retroactively for the whole fleet.
Pull-basedA NUC that was offline self-updates when it reconnects — no need for your Mac to be on.

What the component does (and deliberately doesn't)

The recipe publish.sh generates only installs files: Greengrass unzips the artifact and robocopy-mirrors it into C:\code\holly, leaving .venv alone (it's locked while the app runs, and uv run re-syncs dependencies at launch anyway). Crucially there is no Run step: Greengrass lives in session 0 and must never launch the browser. The watchdog keeps launching from the same folder it always has and simply picks up the new code.

the recipe publish.sh generates — com.holly.app
{
  "ComponentName": "com.holly.app",
  "ComponentVersion": "1.0.1",   // auto-bumped each release
  "Manifests": [{
    "Platform": { "os": "windows" },
    "Lifecycle": {
      "install": "robocopy <unzipped artifact> C:\code\holly /MIR /XD .venv"
      // no "run" — the GUI launch belongs to watchdog.ps1 in holly's session
    },
    "Artifacts": [{ "URI": "s3://holly-greengrass-artifacts/…/holly.zip", "Unarchive": "ZIP" }]
  }]
}

The new code lands on disk as soon as each device pulls the deployment. During kiosk hours, kill the running app once (taskkill over SSH, or close it via VNC) — the watchdog relaunches it on the new code within seconds. Otherwise it's picked up at the next 08:30 launch.

One-time AWS setup Before the first publish: create the artifact bucket (holly-greengrass-artifacts) and grant the devices' GreengrassV2TokenExchangeRole read access to it — the role the installer creates cannot read S3 by default, and without this every deployment fails at download. Exact commands are at the top of publish.sh.
How it meets the local side Greengrass delivers the code; watchdog.ps1 draws it. Clean seam. The tailnet (SSH + VNC) stays on every NUC as the manual/emergency path — it costs nothing to keep.
08

Watch it, and alert

Two layers of "keep it running," because they catch different failures — plus one honest caveat about what Greengrass can and can't see on this fleet.

Session-level (watchdog.ps1) — already built

The app process belongs to the watchdog, not to Greengrass, so crash/hang recovery is already handled locally in holly's session — the layer a session-0 service can never reach.

Device-level (Greengrass)

Because the app isn't a Greengrass-run process, Greengrass reports device connectivity and deployment health, not app state: NUC offline, deployment failed, core device unhealthy. Wire CloudWatch alarms (or Fleet Hub) on those and you get paged when a NUC drops off, instead of discovering it on a VNC check.

Closing the gap: a heartbeat component

To get app state into the cloud too, add one tiny Greengrass component that runs fine in session 0: every minute, read the tail of C:\Watchdog\qtbrowser-watchdog.log (or a status file the watchdog writes) and publish an MQTT heartbeat to IoT Core. Alarm on missed heartbeats and on restart-storm log lines. That gives you "the browser has been crash-looping on HOLLY-002 since 09:14" without the GUI ever leaving the local session.

Minimal-cloud option If you skip Greengrass entirely, the same result is a simple heartbeat POST from the watchdog to any endpoint you own, plus Tailscale's own device-offline signal. The display/schedule layer doesn't change at all.
09

Provisioning a new NUC, end to end

The README in pc-setup is the authoritative step-by-step; this is the same flow with the Greengrass and watchdog steps slotted in. Only the PC number changes per device.

  1. Windows OOBE, offline

    No wifi at any point. Offline account, username holly, password holly. (OOBE\BYPASSNRO via Shift+F10 if Windows blocks the offline path.) Then plug in ethernet at the desktop.

  2. Run setup.ps1
    Terminal (admin)
    Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass
cd C:\Users\holly\Downloads
.\setup.ps1 -PCNumber 004 -AuthKey "tskey-auth-..." `
  -AwsAccessKey "AKIA..." -AwsSecretKey "..."

    15–30 minutes. This is everything: provisioning, watchdog registration (§21), and Greengrass enrolment as holly-004 (§22 — the AWS keys are used for enrolment then cleared). Reboot when asked, then run Windows Update to completion.

  3. Verify it's on both consoles

    From your Mac: holly-004 appears in the Tailscale admin console tagged holly-pc (and VNC/SSH work over the tailnet IP), and shows Healthy under Greengrass → Core devices.

  4. Confirm the app arrived, and test it

    Joining holly-fleet pulls the fleet's current deployment automatically — C:\code\holly appears within a few minutes (or run ./publish.sh for a fresh release). Then a manual uv run holly in C:\code\holly\projects\holly-local to confirm it runs end-to-end.

  5. OBS + devices

    OBS first-run wizard → "I will only be using the virtual camera". Device Manager → disable built-in mics, speakers, and webcam.

  6. Reboot test & label

    Restart. Expected with zero intervention: holly's desktop appears (no login prompt), the watchdog starts and — if inside kiosk hours — launches the app, OBS virtual camera is active, and the device shows online in both Tailscale and Greengrass. Label the box HOLLY-004.

10

Before you commit

Scale fitGreengrass is adopted from the start as a deliberate "have it before we need it" bet: its wins here are versioned releases, offline self-update, and alerting. Staged rollouts are deliberately not enabled (releases are rare — most change happens in the cloud); see 07 for the one-block upgrade if that ever flips.
Windows ≠ LinuxGreengrass is happiest on Linux; on Windows it's supported but fiddlier. If these could ever become Linux devices, the whole stack lightens and balena becomes a strong, purpose-built alternative worth pricing against AWS.
The hard boundaryNo tool removes the session-0 rule. The visible, scheduled, monitor-2 launch is local under every architecture — on this fleet that's watchdog.ps1. Greengrass only ever owns the delivery and monitoring layer around it.
Validate onceSections §21/§22 of setup.ps1 and publish.sh are new and untested on hardware — run the full flow on one NUC end-to-end (enrol → publish → code lands → watchdog launches it) before doing the rest of the fleet.
Docs driftConfirm current Greengrass v2 Windows feature support, installer flags, and recipe syntax in AWS's docs — these shift between releases (e.g. the Corretto winget package id and the PsExec credential step).
